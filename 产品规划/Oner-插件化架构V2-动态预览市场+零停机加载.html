2 零停机热加载机制

所有插件启用/禁用/更新/卸载都不触发应用重启，运行时直接接管。

前端：动态 import + 路由热替换

// frontend/src/kernel/PluginManager.js
class PluginManager {
  async load(pluginId) {
    // 1. 检查本地缓存
    const cached = await this.cache.get(pluginId)
    if (cached) return this.activate(cached)

    // 2. 动态 import（带 hash 防缓存）
    const url = `/plugins/${pluginId}/index.js?t=${Date.now()}`
    const mod = await import(/* @vite-ignore */ url)

    // 3. 注册到运行时
    this.registry.set(pluginId, mod.default)
    this.eventBus.emit('plugin:loaded', pluginId)
    return mod.default
  }

  async unload(pluginId) {
    // 1. 调用插件的 onUnload 钩子
    await this.registry.get(pluginId)?.onUnload?.()

    // 2. 移除路由、清理副作用
    this.router.removePluginRoutes(pluginId)
    this.slotRegistry.unregister(pluginId)
    this.eventBus.off(pluginId)

    // 3. 卸载 ESM 模块（Vite HMR boundary）
    await import(`/plugins/${pluginId}/?unmount`)
    this.registry.delete(pluginId)
  }
}

后端：进程内动态挂载

// backend/src/kernel/PluginManager.js
class BackendPluginManager {
  async mount(pluginId) {
    // 1. require 插件入口（带缓存失效）
    delete require.cache[require.resolve(`/plugins/${pluginId}/index.js`)]
    const plugin = require(`/plugins/${pluginId}/index.js`)

    // 2. 执行迁移
    await this.migrator.run(plugin.migrations)

    // 3. 挂载路由到 Express（不重启 server）
    plugin.routes.forEach(r => {
      this.app.use(r.path, r.handler)
    })

    // 4. 通知前端
    this.ws.broadcast({ type: 'plugin:mounted', pluginId })
  }
}

AI Bridge API（暴露给所有插件）

// 插件中调用 AI 能力
export const MemoPlugin = {
  async onMount(ctx) {
    // 注册 AI 快捷指令
    ctx.ai.registerAction({
      id: 'memo:summarize',
      label: '总结本周备忘',
      icon: '📋',
      scope: ['memo-page'],
      handler: async () => {
        const memos = await ctx.api.get('/memo/recent?range=week')
        return ctx.ai.summarize(memos, {
          template: 'weekly',
          tone: 'concise'
        })
      }
    })

    // 监听页面分析请求
    ctx.ai.onAnalyze('memo-page', async (pageData) => {
      return {
        insights: [
          '本周共创建 12 条备忘',
          '3 条与项目A相关，建议归类',
          '有 1 条超过 30 天未回顾'
        ],
        actions: [
          { id: 'archive-old', label: '归档旧备忘' },
          { id: 'tag-suggest', label: '推荐标签' }
        ]
      }
    })
  }
}

页面分析的实现机制

📊 工作原理：每个页面在挂载时通过 usePageContext() hook 把页面元信息（路径、可见组件、当前数据快照）注册到 AI Bridge。用户点"分析"时，AI 拿到上下文，用插件注册的 onAnalyze handler 拉取专属分析。插件不注册的页面，AI 给出通用分析。

★ 我的补充建议（你的方向我补全这些）

除了你提的"动态UI预览准入"+"零停机"，还有 4 个容易忽略的点：

① AI 能力 vs AI 插件

AI 助手（输入栏、页面分析）放内核，AI 功能（生成、洞察）放插件。这样所有插件开箱即用 AI 能力，不需重复集成。

② 插件版本管理

每个插件独立 semver，内核版本约束（要求 plugin.kernel: "^1.0"）。更新时双版本共存，用户一键迁移数据。

③ 离线/在线双模式

插件包下载到本地后完全离线运行。市场页只是发现机制，安装后无需联网。

④ 开发模式 vs 用户模式

oner dev 命令启动开发态，支持热重载+断点调试。生产态严格权限控制。

还需要确认的 3 个决策点

1. 内置插件 vs 全部可选？ 推荐：Notes 内置（Oner 的核心），其他全部可选可卸载。卸载 Notes 的场景几乎不存在。
2. 插件分发方式？ 三选一：① 本地文件夹（开发者自测） ② HTTP 仓库（团队内网） ③ 官方市场（公共）。建议全部支持，优先级 ① ② ③。
3. 跨插件数据共享？ 通过 EventBus 通信，不直接访问彼此存储。需要数据时通过内核 API 申请，权限审计。
